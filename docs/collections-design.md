# 컬렉션 (Collections) — 설계 v0

> **컬렉션 = 네가 읽고 생각한 것을 *주제로 엮는 채널.*** 글·하이라이트·노트를 **연결**해 담고,
> 같은 것이 여러 컬렉션에 동시에 걸린다(맥락이 겹치는 게 Are.na의 마법). 큐레이션의 단위이자
> 발견의 단위 — 하나로 노트·발견 두 개를 푼다.

이 문서는 AGENTS.md **§0 제품 헌법**("작고 깊게 사랑받는 것 / connect not broadcast")을
처음으로 *기능*으로 옮기는 설계다. 모든 결정은 §0를 통과한다.

## §0 헌법 매핑 (이게 진짜 헌법을 구현하나)

| §0 원칙 | 컬렉션에서의 구현 |
| --- | --- |
| connect, not broadcast | 동사가 "연결". 리포스트의 kurl판 = "내 컬렉션에 연결했다"(재맥락화) |
| 바깥은 조용히 | 컬렉션에 공개 좋아요·팔로워 수 **없음**. "누가" 연결했는지는 보이되 "몇 명"은 안 보임 |
| 안은 풍부히 | 큐레이터 *본인*은 자기 컬렉션 도달(조회·구독)을 사적 대시보드로 봄 |
| 발견 = 큐레이터 | *"민지가 이 글을 '느린 사고'에 연결했어요"* 식으로 흐름. 알고리즘 랭킹 ❌ |
| 점수판 = 깊이 | 컬렉션은 바이럴이 아니라 큐레이션의 깊이를 보상 |

## 확정된 결정 (v0)

1. **라이브러리 vs 컬렉션** — 라이브러리 = 전부 담은 *평면 저장통*(북마크·하이라이트).
   컬렉션 = 그 위의 *주제별 채널*. 컬렉션은 라이브러리를 대체하지 않는다. 북마크 = 빠른 저장,
   연결 = 의도적 배치.
2. **연결 대상** — *아무 글이나* 바로 연결 가능(내가 저장한 것만이 아님). 여러 작가 글을
   가로지르는 큐레이션이 컬렉션의 핵심이라서.
3. **"왜 연결했나" 한 줄** — MVP 포함. 단 선택사항(per-connect). 이게 단순 북마크와 컬렉션을
   가르는 영혼.
4. **공동 큐레이션**(여러 명이 한 컬렉션) — Phase 2.

## 무엇을 연결하나 (블록 타입)

- **글** — 누구 글이든
- **하이라이트** — 글 속 한 구절(이미 1급 객체, 저장+공개귀속 존재)
- **노트** — 내 생각. 독립 노트 또는 글/하이라이트에 붙은 노트
- 외부 링크·URL 임포트 — Phase 2

## "연결" 동사 + 한 줄 이유

어디서든(글 읽다 / 하이라이트 보다 / 노트 쓰다) → **연결** → 컬렉션 고르기(있는 것 또는
새로 만들기, **멀티선택** — 하나가 여러 컬렉션에). 연결할 때 *"왜 연결했는지" 한 줄*을 달 수
있다 — *"이 문단이 헥사고날의 반례"* 같은. 이 한 줄이 큐레이션의 목소리다.

## 조용한 발견 모델

- 컬렉션 공개 범위: **비공개 / 링크공유 / 공개**
- 공개 화면에 **수치 0**(좋아요·팔로워 수 비표시)
- 발견 = 내가 *따르는 큐레이터/컬렉션*의 새 연결이 조용히 흐른다 — 사람의 *외침*이 아니라
  *큐레이션*을 따라가는 것
- 발견 덱(릴스형)엔 큐레이터 연결이 흐른다 → "발견 단위가 무겁다(장문 강제)" 문제를 여기서
  같이 푼다. 릴스에 맞는 가벼운 단위 = 노트·하이라이트·연결

## 데이터 모델 (개념 — 백엔드 엔티티 필요, iOS는 소비자)

```
Collection
  id, ownerId, title, description?, visibility(private|unlisted|public),
  createdAt, updatedAt

Connection (블록)
  id, collectionId, blockType(post|highlight|note), refId,
  why?(한 줄 이유), position, connectorId, connectedAt
```

- **멀티멤버십**: 같은 refId 가 여러 Connection 으로 여러 컬렉션에 걸린다 ← 핵심.
  `Connection` 은 (collection × block) 다대다 조인이다.
- `blockType + refId` 는 다형 참조(post/highlight/note 각 테이블 가리킴).
- 백엔드 슬라이스 규율(헥사고날): 컬렉션이 post·highlight·note 를 직접 의존하면 도메인 사이클.
  cross-domain 은 common.* 중립 포트(예: `ConnectableBlockReader`)를 경유한다.

## IA — 컬렉션은 발견 탭의 심장 (2026-06-15 결정)

컬렉션은 §0 "발견 = 큐레이터의 연결을 따라가기"의 *바로 그 구현*이다 → 주변 기능이 아니라
앱의 심장. 계정 → 서재 3뎁스에 묻어두는 건 헌법과 모순. **발견 탭을 컬렉션·연결 중심으로
재편한다.**

- **발견 탭 = 연결 그래프 홈.** 1차 표면 = *큐레이터 연결 흐름*("민지가 이 글을 '느린 사고'에
  연결했어요" + 왜 한 줄). 내가 따르는 큐레이터/컬렉션의 새 연결이 조용히 흐른다(알고리즘 ❌).
- **기존 릴스 덱은 한 모드로 흡수** — "읽기 모드"(선택 없이 몰입해 바로 읽기)는 발견 안의
  보조 모드로 남는다. 두 모드 다 §0: 흐름 = 큐레이터 주도 브라우즈, 읽기 = 몰입 독서.
- 계정 → 서재의 "컬렉션" 행은 *내 컬렉션 관리* 진입으로 유지(발견 = 남 둘러보기, 계정 = 내 것).
- 탭바는 5개 유지(검색·글쓰기 그대로). 6번째 탭 ❌(§1 5탭 규칙).

## 화면 (MVP)

1. **발견 = 큐레이터 연결 흐름** — 연결 이벤트 카드(큐레이터 + 블록 + 컬렉션 + 왜 한 줄) +
   "읽기 모드"(릴스 덱) 진입
2. **내 컬렉션 목록** — 계정/라이브러리 안
3. **컬렉션 상세** — 담긴 블록(글 카드 · 하이라이트 인용 · 노트 행 섞임) + 각 연결의 한 줄 이유
4. **"연결" 시트** — 글/하이라이트/노트에서 컬렉션 고르기(+ 새로 만들기 · 멀티선택 · 이유 한 줄)
5. **공개 컬렉션 페이지** — 공유 가능

디자인 언어(§1)는 종이 세계: 컬렉션 상세는 카드/인용/행의 종이 문법, 크롬(연결 시트·생성 FAB)만
유리. 공개 수치 비표시는 §0 + §10 색·절제 규율과 정렬.

## 기존 것과의 경계

- **시리즈** = 작가 *자기 글의 선형 출판물*(만들기). **컬렉션** = 누구나, 여러 작가를
  *가로지르는* 연상적 큐레이션(잇기). 닮았지만 다른 것.
- **노트** = 컬렉션에 연결되며 집·맥락을 얻는다 → 노트 외로움(순수 독서도구의 함정) 해결.
- **하이라이트** = 이미 라이브러리에 저장됨. 주제 컬렉션에 연결은 자연스러운 확장.
